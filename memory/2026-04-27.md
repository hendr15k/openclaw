## HappyBlue Bluetooth-Fix — `DeviceListActivity.java`

**Problem:** Adapter werden nicht in der Bluetooth-Geräteliste angezeigt

**3 Bugs gefunden und gefixt:**

1. **Discovery-Filter** (Zeile ~52): `if (device.getBondState() != 12)` — bonded Geräte (BondState=12) wurden aus Suchergebnissen gefiltert → **Fix:** Filter entfernt, Deduplizierung per `getPosition()` stattdessen

2. **Permission-Race Condition Android 12+** (Zeile ~166): `getBondedDevices()` wurde sofort aufgerufen bevor `BLUETOOTH_CONNECT` Permission da war → SecurityException wurde gefangen → Liste blieb leer → **Fix:** `loadBondedDevices()` Methode mit try/catch, wird NUR nach Permission-Grant aufgerufen (in `onRequestPermissionsResult`)

3. **Bonded Devices erschienen nicht vor Discovery**: `doDiscovery()` startete Discovery ohne vorherige bonded Devices → **Fix:** `loadBondedDevices()` wird JETZT VOR `startDiscovery()` in `doDiscovery()` aufgerufen

**APK:** `happyblue-bt-fix.apk` (12 MB, Debug-Signatur)
**Änderung:** 3 separate Fixes in einer Datei

## HappyBlue Foreground-Service Crashes — Nachmittagssession (14:45–15:58)

**Hendrik schickte 3 Crash-Reports nacheinander:**

### Crash 1: `SecurityException: startForeground requires FOREGROUND_SERVICE`
- **Root Cause:** Android 9+ (API 28) braucht `android.permission.FOREGROUND_SERVICE` + `foregroundServiceType`
- **Fix:** Permission + `foregroundServiceType="connectedDevice"` zum Service im Manifest

### Crash 2: `IllegalArgumentException: PendingIntent requires FLAG_IMMUTABLE`
- **Root Cause:** Android 12+ (API 31) erzwingt `FLAG_IMMUTABLE` oder `FLAG_MUTABLE` bei PendingIntents
- **Fix:** 8 PendingIntents in `BluetoothService.java` + `UniversalBluetoothService.java` mit `PendingIntent.FLAG_IMMUTABLE` (Wert `134217728 + 67108864 = 201326592`)
- **Commits:** `76513fc`, `ce74d64`, `d8d8d24`

### Crash 3: `CannotPostForegroundServiceNotification: Bad notification for startForeground`
- **Root Cause:** App nutzt `support library 25.4.0` (keine `NotificationCompat.Builder(context, channelId)` Methode), `Notification.Builder` ohne Channel → ab Android 8.0 → BadNotification Exception
- **Fix (multi-step):**
  1. `NotificationChannel` in `HappyBlue.onCreate()` erstellen (VOR `startForeground()`)
  2. `BluetoothService.showConnectedNotification()` auf nativen `Notification.Builder` umgeschrieben (statt NotificationCompat.Builder mit setChannel)
  3. `UniversalBluetoothService.showConnectedNotification()`同样 auf nativen `Notification.Builder`
  4. String-Referenzen hardcoded (R.string aus common library nicht erreichbar in context)
- **Commit:** `7a1e6d2`
- **APK:** `build-20260427-135420-7a1e6d2`

### BT Log Button — Feature Request von Hendrik
- **Was:** "Export BT Log" Button im MainMenu-Optionsmenü (⋮)
- **Output:** `BT_Log_YYYYMMDD_HHmmss.txt` in Downloads/
- **Inhalt:** Adapter-Name/MAC, gebondete Geräte, gespeicherte MAC, Permission-Status, letzte 100 HappyBlue-Logcat-Zeilen
- **Commits:** `d8d8d24` + `7a1e6d2`

## Learnings (Session 2)

- LRN-20260427-005: `NotificationCompat.Builder` ab support library 26.1.0 — vorher KEIN Channel-Parameter, nur `Notification.Builder(Context)` + `NotificationChannel` manuell
- LRN-20260427-006: Support library `NotificationCompat.Builder.setChannel()` ist ein NO-OP — ignoriert wird. Erst native `Notification.Builder` mit Channel-ID funktioniert
- LRN-20260427-007: R.string-Referenzen aus common-library können nicht direkt in app-context genutzt werden → hardcoded Strings als Workaround
- LRN-20260427-008: `PendingIntent.FLAG_IMMUTABLE` Wert = `67108864`, `FLAG_MUTABLE` = `33554432` (API 31+)
- LRN-20260427-009: `foregroundServiceType="connectedDevice"` ist ab API 31 verfügbar, auf API 33 Target → kompatibel
- LRN-20260427-010: Crash-Pattern: 3 aufeinanderfolgende Crashes nach BT-Connect-Fix → jeder Fix deckte nächste Requirement auf (Permission → PendingIntent → NotificationChannel)

### Problem: Pairing funktioniert nicht

**Symptom:** Handy findet `obdzwei`, aber Verbindung schlägt fehl. Keine Pairing-Attempts in Logs.

**Ursachen identifiziert:**
1. **Kein Bluetooth Agent** — Pairing-Requests wurden abgelehnt ("No agent available")
2. **SDP Record fehlte** — Serial Port Profile nicht advertised, Handy fand RFCOMM-Channel nicht
3. **ngrok Free instabil** — SSH-Sessions brechen ständig ab, Portforwarding zu Laptop nicht stabil

**Lösungen implementiert:**
- SDP Record: `sudo sdptool add --channel=1 SP` (Serial Port auf RFCOMM Channel 1)
- D-Bus Agent: `/home/hendrik/bt_agent_dbus.py` mit NoInputNoOutput — auto-acceptet alle Pairings
- Python ELM327 Server: `/home/hendrik/elm327_server.py` — minimaler OBD-Responder

**Aktueller Stand (22:23):**
- SDP Record ✅ (`Service Name: Serial Port, RFCOMM Channel 1`)
- Agent ✅ (via bluetoothctl + expect fallback)
- Discoverable/Pairable ✅ (Timeout=0)
- Server gestartet auf RFCOMM
- **Aber:** Car Scanner verbindet sich nicht — evtl. neues Pairing nötig

**Wichtig fürnächste Session:**
- ngrok Free ist zu instabil für komplexe Multi-Step BT-Setup-Sessions
- Script `bash ~/setup_obd.sh` muss manuell auf Laptop ausgeführt werden
- Nach jedem Reboot: ngrok neu starten + Scripts neu ausführen
- Car Scanner sucht explizit nach "OBDII" / "ELM327" Device-Typ — evtl. Class of Device ändern
- SDP "Service Name: Serial Port" ist korrekt für Car Scanner

### LRN-20260427-011: RFCOMM `ss` nicht sichtbar
- `ss -tlnp | grep rfcomm` zeigt NUR TCP/UDP sockets, nicht RFCOMM
- RFCOMM sockets nur in `/proc/net/rfcomm` sichtbar
- Oder: `sudo cat /proc/net/rfcomm`

### LRN-20260427-012: Ngrok Free = unstable
- Sessions schlafen nach Inaktivität → Portforwarding tot
- Neu starten mit `ngrok tcp 22` jedes Mal
- Bessere Alternative für stabilen Zugang: IPv6 oder VPN

### LRN-20260427-013: Bluetooth Agent Object Path

- Korrekter Pfad: `/org/bluez/agent` → funktioniert
